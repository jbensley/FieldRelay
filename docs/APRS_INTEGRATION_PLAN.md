# APRS / aprs.fi Integration Plan

APRS-IS outbound transmission is now partially implemented. Remaining items in this document are follow-up design notes for validation, aprs.fi read-side lookup, and any deeper APRS features.

## Key Finding

`aprs.fi` is primarily a read/query API for data already collected by aprs.fi. It should be used for looking up APRS stations, recent positions, weather data, or messages. It is not the right API for transmitting this app's live position.

For outbound APRS location transmission, the app needs an APRS-IS client path. aprs.fi can then be used as a read-side verification/display source after packets reach APRS-IS and are ingested by aprs.fi.

## Required User Settings

Add a dedicated APRS section in Settings:

- Enable APRS transmission: boolean, default off.
- APRS callsign + optional SSID, e.g. `N0CALL-7`.
- APRS-IS passcode is entered explicitly alongside the callsign and stored as a separate setting.
- APRS server/port and symbol table/code are advanced settings. Defaults should be hidden for normal use unless the user explicitly expands advanced APRS settings.
- APRS comment/status text, default `FieldRelay Android`.
- APRS transmit interval currently uses the shared beacon interval; a separate interval remains a future option.
- Optional aprs.fi API key for read-side lookup/verification remains future work.
- Optional aprs.fi lookup callsign if different from transmit callsign remains future work.

Important: APRS-IS passcodes are tied to amateur radio callsigns. The UI should clearly state that APRS transmission is for licensed amateur radio operators using their own callsign. FieldRelay requires the operator to enter their APRS-IS passcode explicitly.

## Data Layer

Extend `SettingsRepository` with APRS preferences:

- `aprsEnabled: Boolean` implemented.
- `aprsCallsign: String` implemented.
- APRS-IS passcode is stored as a separate setting and is not generated by the app.
- `aprsServer: String` implemented.
- `aprsPort: Int` implemented.
- `aprsSymbolTable: String` implemented.
- `aprsSymbolCode: String` implemented.
- `aprsComment: String` implemented.
- `aprsTransmitIntervalMs: Long` not implemented; shared beacon interval is used.
- `aprsFiApiKey: String` not implemented.
- `aprsFiLookupCallsign: String` not implemented.

Keep APRS settings separate from Spotter Network settings so either service can be enabled independently.

## Network Components

### aprs.fi Read Client

Add package:

```text
aprsfi/
  AprsFiApi.kt
  AprsFiModels.kt
  AprsFiRepository.kt
```

Responsibilities:

- Query `https://api.aprs.fi/api/get`.
- Support at least `what=loc`.
- Include `apikey`, `format=json`, and callsign `name`.
- Set a useful HTTP `User-Agent` with app name/version.
- Parse success/failure shape:
  - `result=ok`
  - `result=fail`
  - `description` for errors.

### APRS-IS Transmit Client

Add package:

```text
aprsis/
  AprsIsClient.kt
  AprsPacketFormatter.kt
  AprsPosition.kt
  AprsTransmitResult.kt
```

Preferred initial transport:

- HTTPS send-only POST to a javAPRSSrvr server supporting send-only HTTP/HTTPS.
- Default server should be configurable, not hard-coded as the only option.
- Body contains login line plus one or more TNC2 packets, unless HTTP Basic or APRS-IS Authorization is used.

Fallback/advanced transport:

- TCP to APRS-IS user-defined feed port `14580`.
- Login line, then send TNC2 packets terminated with CR/LF.

## Packet Formatting

Start with uncompressed position packets for readability and lower implementation risk.

Formatter inputs:

- callsign/SSID
- latitude/longitude
- altitude
- speed/course
- symbol table/code
- comment
- timestamp if desired

Formatter output:

```text
CALL-SSID>APRS,TCPIP*:!DDMM.mmN/DDDMM.mmW>comment
```

Exact symbol and optional course/speed/altitude formatting should be verified against APRS protocol references before implementation.

## LocationService Integration

Do not create a second location loop. Reuse the existing `LocationService` fixes:

1. `LocationService` receives a GPS fix.
2. It posts to Spotter Network if Spotter reporting is enabled.
3. It posts to APRS-IS if APRS transmission is enabled and APRS settings are valid.
4. It updates telemetry with separate last-result fields for Spotter Network and APRS.

This likely requires changing `TelemetryState` from one generic `lastError` to service-specific status fields:

- `lastSpotterReportAt`
- `lastSpotterError`
- `lastAprsTransmitAt`
- `lastAprsError`

## UI Changes

Settings:

- Add APRS settings section.
- Validate callsign format locally.
- Add help text explaining amateur-radio callsign/passcode requirements.
- Gate APRS server/port and symbol table/code behind an advanced-settings toggle.
- Add optional aprs.fi API key field for lookup/verification.

Home:

- Show Spotter Network status and APRS status separately.
- Start/stop button can continue controlling the shared location service.
- If APRS is enabled but invalid, show a clear settings warning.

Diagnostics:

- Show APRS enabled/on/off.
- Show callsign configured yes/no.
- Show last APRS transmit time/error.
- Show aprs.fi verification result if implemented.

## Testing Plan

Pure JVM tests:

- Callsign validation.
- Standalone APRS-IS passcode utility outside the Android app.
- Decimal-degree to APRS coordinate conversion.
- TNC2 position packet formatting.
- APRS-IS login line formatting.
- aprs.fi response parsing.
- Transmit throttling logic.

Integration/manual tests:

- Send to APRS-IS test path with a valid licensed callsign.
- Confirm packet appears via aprs.fi lookup.
- Confirm failures surface separately from Spotter Network failures.
- Confirm airplane-mode recovery does not retry too aggressively.

## Open Questions

- Which APRS-IS server/transport should be the default: HTTPS send-only, TCP `14580`, or both?
- Should APRS transmission use the same interval as Spotter Network, or a separate interval?
- Should APRS continue when Spotter Network reporting is off, or should both be controlled by one shared beacon toggle?
- What default APRS symbol should FieldRelay use?
- Should V2 include read-only aprs.fi lookup before outbound APRS-IS transmission?
